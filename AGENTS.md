# AGENTS.md

## 1. 项目定位

本项目是一个 Windows 本地桌面工具，用于个人麻将学习和复盘。它通过手动框选雀魂游戏画面，周期性截取指定区域，识别用户自己的手牌、摸牌、宝牌指示牌和副露区，并给出以牌效为主的向听数、有效牌和切牌排序。

项目边界必须保持清晰：

- 不自动点击、出牌或操作游戏窗口。
- 不接入账号、网络协议或游戏内部数据。
- 不实现规避检测、隐藏式实战辅助等功能。
- 识别和分析均在本机完成，用户截图、配置和模板不提交到仓库。

## 2. 当前技术栈

- Python 3.12 及以上；若 Python 3.14 的图像依赖安装异常，优先使用 Python 3.12。
- PySide6：桌面界面、计时器、区域框选。
- mss：按物理屏幕坐标截图。
- OpenCV、NumPy、Pillow：图像裁剪、模板匹配和诊断图保存。
- unittest：当前测试框架。

安装与启动：

```powershell
python -m venv .venv
.\.venv\Scripts\python -m pip install -r requirements.txt
.\.venv\Scripts\python run_app.py
```

运行测试：

```powershell
.\.venv\Scripts\python -m unittest
```

截至 2026-07-15，测试共 51 项，全部通过。

## 4. 核心数据与约定

### 4.1 牌名

标准牌名为：

- 数牌：`1m` 至 `9m`、`1p` 至 `9p`、`1s` 至 `9s`。
- 字牌：`east south west north white green red`。
- 赤五模板可命名为 `5mr 5pr 5sr`；进入牌效计算前分别归一化为 `5m 5p 5s`。

模板文件名的 stem 就是识别结果中的牌名。当前本机模板已经补齐 34 种普通牌及 3 种赤五，共 37 个模板，但这些用户生成图片不会进入 Git。

### 4.2 配置

实际配置位于 `data/config.json`。主要字段：

- `game_region`：物理屏幕坐标，包含 `left/top/width/height`。
- `layout.*_region`：相对于 `game_region` 的 0 至 1 比例坐标。
- `hand_tile_count`：手牌区当前可见暗牌张数。
- `draw_tile_count`：独立摸牌区张数，只允许 0 或 1。
- `meld_tile_count`：副露区当前可见牌张数。
- `open_meld_count`：已完成副露组数，范围 0 至 4；牌效分析直接使用此值。
- `melds`：结构化副露组列表；每组记录 `kind`，每张可见牌记录相对副露总区的 `region`、`orientation` 和 `stack_level`。
- `recognition.threshold`：最低匹配置信度，默认 0.78。
- `capture_interval_seconds`：连续识别间隔，默认 0.75 秒。

常用牌数配置：

- 闭门摸牌后：`13 1 0 0`。
- 一组副露后待切：`10 0 3 1`。
- 一组杠、杠后补牌：`10 1 4 1`（副露区应计入杠的 4 张可见牌；叠放样式仍需按画面准确框选）。
- 两组副露后待切：`7 0 6 2`。

### 4.3 识别结果

`RecognitionResult` 包含：

- `hand`：手牌区识别结果。
- `draw`：独立摸牌，未配置时为 `None`。
- `dora_indicators`：宝牌指示牌列表。
- `meld_tiles`：副露区从左到右的可见牌。
- `melds`：逐组、逐张的结构化识别结果，可保留单张未知状态和最佳候选。
- `confidence`：所有已识别区域匹配分数的平均值。
- `matches`：逐张牌的名称和分数。

手牌、摸牌或宝牌低于阈值时，整次识别会抛出 `RecognitionError`；副露低于阈值时记录警告并继续牌效分析。

## 6. 实际调试中学到的知识

### 6.1 分辨率与 DPI

雀魂内部设置的 1920x1080 与 Windows 桌面缩放是两个维度。Windows 125% 缩放会让 Qt 返回逻辑坐标，而 mss 使用物理像素；如果不转换，1920x1080 画面可能被错误记录成约 1536x864，牌区随之偏移。当前入口先启用 DPI awareness，框选器再乘 `devicePixelRatio()`。

修改框选或截图代码时必须同时验证：

- Windows 缩放 100% 和 125%。
- 游戏分辨率 1920x1080。
- `game_region` 保存尺寸是否与实际游戏画面一致。
- `full_capture_regions.png` 中彩色框是否贴合真实牌区。

### 6.2 框选精度决定模板质量

模板和实时识别必须使用相同的游戏分辨率、窗口尺寸、皮肤及近似牌区边界。框选应覆盖完整牌面，但尽量少带桌面背景、阴影和相邻牌。只看区域位置正确还不够，逐张切分边界也必须正确。

诊断顺序：

1. 看 `full_capture_regions.png`，确认大区域位置。
2. 看 `hand_*.png`、`draw_*.png`、`dora_*.png`、`meld_*.png`，确认每张图只含一张完整牌。
3. 若切分正确但分数低，检查模板是否来自相同显示条件。
4. 再考虑微调阈值；不要先用大幅降低阈值掩盖区域或模板问题。

### 6.3 工具窗口与闪烁

如果工具窗口覆盖游戏画面，它会进入截图并污染识别。当前仅在窗口与 `game_region` 相交时隐藏 180 ms 后截图。连续闪烁通常来自窗口反复隐藏和显示，不是因为副露后暗牌数量变成 11/8；将工具移到游戏区域外可避免隐藏。

### 6.4 副露后的牌数状态

副露后正常情况下不会产生独立摸牌：鸣牌完成后直接打出一张，因此常见配置是 `10 0 3 1` 或 `7 0 6 2`。杠后会补牌，才可能出现独立摸牌区为 1。四个数字依次表示手牌区张数、摸牌张数、副露可见张数和副露组数；牌数配置必须描述当前画面真实可见牌。

### 6.5 模板与赤五

赤五画面应使用独立模板 `5mr/5pr/5sr`，否则颜色和图案差异可能降低置信度。牌效层会把赤五归一化成普通五，这对向听和进张正确，但会丢失赤宝牌身份。未来做保留赤五偏好或打点时，必须在分析数据中另外保存赤牌标记。

### 6.6 副露识别失败的降级策略

副露区受横置、叠放和组间空隙影响，等宽切分的置信度不应决定核心牌效是否可用。当前仅手牌、摸牌和宝牌识别失败会抛出整帧错误；副露单张低于阈值时，该张记录未知状态与最佳候选，其他成功副露牌继续保留，并使用配置中的 `open_meld_count` 进行牌效分析。副露失败匹配不计入置信度。

### 6.7 结构化副露坐标

结构化副露的每个牌槽使用相对整个 `meld_region` 的 0 至 1 坐标。横置牌必须在匹配前按 `rotated_cw` 或 `rotated_ccw` 旋转归一；加杠等叠放可用相互重叠的牌槽和不同 `stack_level` 表达。核心平均置信度只统计手牌、摸牌和宝牌，副露按组单独计算，避免副露布局变化导致核心置信度不可比较。

## 8. 开发与修改约定

- 保持现有 `app / recognizer / mahjong` 分层；UI 不直接实现图像算法或向听算法。
- 新增结构化数据优先使用 dataclass，避免用松散字典在模块间传递。
- 配置新增字段时必须给出默认值，并保证旧 `data/config.json` 可加载。
- 所有屏幕绝对坐标使用物理像素；牌区内部位置使用相对游戏区域的比例。
- 图像颜色约定：mss 输出经 Pillow 转为 RGB；OpenCV 模板内部转为 BGR。
- 不提交 `data/config.json`、`data/debug/`、用户截图或 `data/templates/*.png`。
- 修改向听或有效牌逻辑必须增加确定牌例测试。
- 修改裁剪、匹配或 DPI 逻辑必须至少运行全部测试，并通过真实截图检查诊断图。
- 不以降低全局置信度阈值作为识别问题的首选修复。
- 不加入自动操作游戏的代码。
